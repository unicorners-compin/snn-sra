## 2. 结果总结 (Result Summary)

### 2.1 核心发现
本次实验模拟了在 10x10 网格拓扑结构下，100 个自主节点处理 300 个随机流量流的动态过程。实验重点观察了**节点应力（Stress）**随迭代次数的演化情况。

### 2.2 关键数据指标
*   **应力增长趋势**：系统最大应力从初始状态的 **7.95** 持续上升，最终在第 180 轮迭代时达到 **26.42**。
*   **瓶颈节点分析**：实验追踪了初始状态下的瓶颈节点（ID: 56）。该节点的应力从 **5.07** 增长至 **24.95**。
*   **收敛特性**：
    *   在 0-100 轮迭代期间，应力呈快速线性增长。
    *   在 100-200 轮迭代期间，增长斜率明显放缓，系统进入高位震荡或准稳定状态（例如迭代 120 与 140 之间出现了微小的应力回落）。

### 2.3 结论
当前的路由自适应参数（$\beta_I=8.0$）未能有效实现负载均衡，反而导致了应力在特定节点（如中心区域节点）的聚集。系统表现出明显的“富者更丰”或“路径依赖”特征，导致局部拥塞加剧。

## 3. 代码分析 (Code Analysis)

### 3.1 核心算法逻辑
源代码实现了一个基于自主代理（Autonomous Agent）的去中心化路由仿真框架，其核心逻辑如下：

1.  **拓扑构建**：利用 `generate_grid_topo(10)` 生成 100 节点的二维网格拓扑，模拟典型的分布式网络环境。
2.  **节点自主性**：每个 `AutonomousNode` 实例拥有独立的决策参数：
    *   `alpha` (0.1): 可能代表学习率或应力衰减因子。
    *   `beta_I` (8.0): 路由切换意愿强度。较高的值意味着节点对应力变化更敏感。
    *   `T_d` (5): 决策延迟或时间窗口。
3.  **仿真循环**：
    *   `sim.run_step(k, traffic_pairs)`：在每一步中，模拟器根据当前各节点的应力状态重新分配或更新流量路径。
    *   **动态追踪**：代码通过 `np.argmax` 锁定初始状态下压力最大的节点，并持续记录其在整个生命周期内的应力演化。
4.  **数据持久化**：通过 `history` 字典记录全局平均应力（`v_s`）、最大应力（`max_s`）、标准差（`std_s`）以及初始/最终状态快照。

### 3.2 关键函数依赖
*   `DecentralizedSimulator`: 负责执行去中心化的路由寻址算法。
*   `AutonomousNode`: 封装了节点级别的应力计算与路由决策逻辑。

## 4. 问题与建议 (Issues & Suggestions)

### 4.1 潜在问题 (Bugs & Bottlenecks)
1.  **应力累积机制风险**：日志显示最大应力持续上升。如果 `AutonomousNode` 内部的应力计算是累加性的而缺乏有效的释放或衰减机制，仿真可能会演变为数值发散，无法反映真实的动态平衡。
2.  **瓶颈节点锁定偏差**：代码在 `k=0` 时锁定了 `peak_node_id`。然而，随着路由自适应的进行，系统的瓶颈节点可能会发生漂移。仅追踪初始瓶颈节点可能无法全面反映后期系统的最薄弱环节。
3.  **随机性样本单一**：`traffic_pairs` 在主循环外生成且固定。这虽然保证了实验的可重复性，但可能导致结果受到特定随机分布的严重影响（例如流量恰好集中在中心区）。

### 4.2 改进建议
1.  **引入应力衰减**：建议在 `AutonomousNode` 中引入类似 $S_{t+1} = (1-\gamma)S_t + \text{new\_load}$ 的机制，以模拟流量完成后的资源释放。
2.  **动态瓶颈追踪**：在 `history` 中增加每轮迭代中 `argmax(current_stresses)` 的记录，对比“初始瓶颈”与“实时瓶颈”的重合度，以评估路由算法的负载转移能力。
3.  **参数敏感性分析**：目前的 `beta_I=8.0` 似乎导致了较强的正反馈。建议对比测试 `beta_I` 在 [0.1, 1.0, 5.0] 不同区间下的表现，寻找实现负载均衡的最佳阈值。
4.  **性能优化**：如果 `NUM_NODES` 扩展到 1000 以上，当前的 Python 循环结构可能成为瓶颈。建议将 `sim.run_step` 中的核心计算逻辑向量化（使用 NumPy）。
